# 03 · 闭环理论(核心)

本章阐述整个系统"逻辑闭环"的原理,分三块:**编码器里程计**、**PID 速度闭环**、**EKF 传感器融合**。

---

## 3.1 为什么开环里程计不行(问题定义)

一种朴素做法(开环里程计)是直接积分**指令速度** `cmd_vel`:

```
x += v_cmd · cos(θ) · dt
```

只要发生以下任一情况,推算位置就与真实脱节且**误差永久累积**:

- 轮子打滑 / 空转(地面、负载变化)
- 电机堵转(重载起步,PWM 给了但没动)
- 电池电压下降,同一 PWM 实际转速变低

里程计一旦漂移:
1. **SLAM** 的 `odom→base_link` 不准 → 扫描匹配约束错误 → 地图重影、闭环失败;
2. **AMCL** 运动模型预测错 → 粒子发散 → 定位丢失;
3. **导航**跟踪的是错误位姿 → 撞墙或原地打转。

结论:**必须让里程计反映"实际动了多少",这就需要轮速编码器。**

---

## 3.2 正交编码器 x4 解码

增量编码器有 A、B 两相,相位差 90°。正转与反转时 A/B 的先后不同,
据此既能测**位移**也能判**方向**。

x4 解码:A、B 的**每个上升沿和下降沿都计数**,一个电周期得到 4 个计数,
分辨率最高。用状态转移表实现(见 `encoder_node.py` 的 `_TABLE`):

```
state = (A<<1)|B           # 4 种状态: 00,01,11,10
合法单步跳变 -> +1 或 -1
非法跳变(丢步/抖动) -> 0   # 抗噪
```

累计计数换算:

```
Δ距离(单轮) = Δticks × meters_per_tick
meters_per_tick = 2π·r / (4·CPR·gear_ratio)
瞬时轮速 = Δ距离 / Δt        # 再过一阶低通滤波抑制量化噪声
```

## 3.3 内层:PID 速度闭环(控制闭环)

**目标**:让每个轮子的实际转速跟上指令转速,与负载/电压无关。

差速运动学把机器人速度 `(v, ω)` 分解到左右轮线速度:

```
v_left  = v − ω·L/2
v_right = v + ω·L/2      # L = wheel_base
```

每个轮子跑一个独立 PID,误差 = 目标轮速 − 编码器实测轮速:

```
e(t)   = v_target − v_measured
u(t)   = Kp·e(t) + Ki·∫e dt + Kd·de/dt        # 带积分限幅
PWM    = 前馈(v_target/v_max·ff_gain) + u(t)   # 前馈加速收敛
```

工程细节(都在 `l298n_motor_node.py` 里实现):

- **前馈 + 反馈**:前馈给一个"大概该多少 PWM",PID 只补差,收敛快、不易积分饱和。
- **积分限幅** `pid_i_max`:防止堵转时积分无限累积(integrator windup)。
- **死区/最小 PWM 提升**:L298N + 减速电机有启动死区,`min_pwm` 把有效输出抬到能真正转动的区间。
- **反馈超时回退**:编码器数据超过 `feedback_timeout`(0.3s)没来,自动退回开环"速度→PWM"曲线,保证不失控。
- **cmd_vel 看门狗**:0.6s 没有新指令 → 停车 + 复位 PID。

> 这一层的产出有两个:①电机真的按指令速度转;②`wheel_speeds_std` 给里程计/网页用的**可信实测轮速**。

## 3.4 外层:编码器里程计(差速正运动学)

`wheel_odometry_node` 用**编码器增量**做精确弧线积分(exact arc integration):

```
d_left  = Δticks_left  × meters_per_tick
d_right = Δticks_right × meters_per_tick
d_center = (d_left + d_right) / 2
d_theta  = (d_right − d_left) / wheel_separation

若 |d_theta| < ε:                       # 近似直线
    x += d_center·cos(θ);  y += d_center·sin(θ)
否则:                                    # 走半径 R 的圆弧
    R = d_center / d_theta
    x +=  R·(sin(θ+d_theta) − sin(θ))
    y += −R·(cos(θ+d_theta) − cos(θ))
θ += d_theta
```

比"直线近似"更准(转弯时不会系统性偏内/偏外)。输出 `nav_msgs/Odometry`,
并**认真填写协方差**:x/yaw 可信(小方差),累积项给中等方差——这是给下一步 EKF 用的权重。

## 3.5 融合层:EKF(估计闭环)

编码器里程计仍有一个天生弱点:**航向角 θ 由左右轮差分推出,长时间累积会慢漂**
(轮径不等、打滑、间隙)。而 IMU 恰好擅长测**姿态/角速度**,不擅长测位移。二者互补:

| 量 | 编码器 | IMU | 融合后 |
|----|--------|-----|--------|
| 前进距离 vx | ✅ 准 | ❌ 加速度积分漂 | 用编码器 |
| 航向 yaw | ⚠️ 慢漂 | ✅ 稳 | 用 IMU 纠偏 |
| 角速度 vyaw | ✅ | ✅ | 互相印证 |

用 `robot_localization` 的 **EKF**(扩展卡尔曼滤波)融合。状态量在 `two_d_mode`
下锁定 z/roll/pitch,只估平面 `[x, y, yaw, vx, vyaw]`。

配置策略(见 `ekf.yaml`):

```
odom0 (编码器):  fuse vx, vyaw            # 提供线速度与角速度
imu0  (IMU):     fuse yaw(绝对), vyaw     # 用绝对航向摁住漂移
```

EKF 两步循环:

```
预测:  用运动模型 + process_noise 把上一时刻状态推到当前
更新:  用每个传感器测量 + 其协方差,按卡尔曼增益加权修正
```

**EKF 输出 `odom→base_link` 这个 TF**,成为全系统唯一权威的位姿来源。
这样即使没有轮式编码器之外的绝对定位,`odom` 系下的位姿也足够平滑、航向不漂,
SLAM 扫描匹配能收敛、AMCL 能锁定——**闭环就此形成**。

## 3.6 一张图看懂"两层闭环"

```
        ┌──────────────── 外层:估计闭环 ────────────────┐
        │                                                │
 cmd_vel│    ┌── 内层:控制闭环 ──┐                        │
   │    │    │                   │                        │
   ▼    │    ▼                   │                        │
 target─┼─► PID ─► PWM ─► 电机 ─► 车动 ─► 编码器 ─┬─► 轮速 ─┘(反馈给PID)
        │                                        │
        │                                        └─► 里程计 ─┐
        │                                                    ├─► EKF ─► 位姿(odom→base_link)
        │                          IMU ────────────────────┘         │
        │                                                             │
        └───────────── SLAM/AMCL 用这个位姿定位、Nav2 用它导航 ◄───────┘
```

- **内层**(几十 Hz):保证"轮子按命令转"。
- **外层**(30 Hz EKF):保证"我知道自己在哪、朝哪"。
- 两层都闭合 → 上层 SLAM/导航才有可信输入。
